## Findings

### 1. Web-saved solar settings are silently dropped on reload

Severity: High

Evidence:

- [solar_ev_controller.py](solar_ev_controller.py:41)
- [web_config.py](web_config.py:190)
- [templates/charger.html](templates/charger.html:152)

`web_config.py` saves several profile fields from the charger page, but `normalize_automation_config()` only preserves keys listed in `AUTOMATION_PROFILE_KEYS`. The current whitelist does not include at least these fields that the form writes:

- `tesla_solar_powerwall_boost_ev_share_pct`
- `tesla_solar_powerwall_high_ev_share_pct`
- `tesla_solar_powerwall_low_ev_share_pct`
- `tesla_solar_after4pm_ev_share_pct`
- `tesla_solar_change_threshold_amps`
- `tesla_solar_change_sustain_down_seconds`
- `tesla_solar_change_sustain_up_seconds`

Impact:

- The UI appears to save these values, but the controller drops them on the next config load/reload.
- Some of the new web-config behavior is therefore not actually persistent.

Possible fix plan:

1. Add every intended profile key to `AUTOMATION_PROFILE_KEYS`.
2. Add a round-trip config test: load -> normalize -> save -> reload must preserve editable profile fields.
3. Make the dashboard/status surface expose the live effective values so UI persistence is easy to verify.

### 2. The charger page regenerates Tesla OAuth state on every GET

Severity: High

Evidence:

- [web_config.py](web_config.py:121)
- [tesla_energy.py](tesla_energy.py:714)

`charger_page()` calls `tesla_monitor.build_authorize_url()` during a normal page render. `build_authorize_url()` always generates a new `oauth_state` and writes it to the token bundle.

Impact:

- Refreshing `/charger` can invalidate an in-flight Tesla authorization started from an earlier page view.
- Multiple open tabs can race and cause `Tesla OAuth state mismatch`.

Possible fix plan:

1. Stop generating OAuth state during page render.
2. Generate state only from an explicit authorize action or dedicated endpoint.
3. Reuse an unexpired pending state when one already exists.

### 3. `/api/status` serves cached charger state whenever cache exists

Severity: High

Evidence:

- [web_config.py](web_config.py:303)

`/api/status` only calls `build_status_report()` when no cache file exists. If cache exists, the route always returns the cached report and labels it as `"source": "cache"`.

Impact:

- The dashboard is not actually doing a live charger read with cached fallback.
- A stopped or unhealthy automation loop can be masked by an old cache file.
- Operator decisions can be made from stale state without an explicit hard failure.

Possible fix plan:

1. Attempt `build_status_report(cfg)` first.
2. Only fall back to `load_status_report_cache()` in the exception path.
3. Keep the current age/stale metadata, but attach it only to real fallback responses.

### 4. Changing the active profile can overwrite another profile with the wrong values

Severity: Medium

Evidence:

- [templates/charger.html](templates/charger.html:111)
- [web_config.py](web_config.py:185)

The form fields are rendered from the current `active_profile`, but on POST the handler writes those submitted values into `requested_profile`, which comes from the dropdown selection.

Example:

- Page loads while `summer` is active.
- User changes the dropdown to `spring` and clicks save.
- The visible `summer` solar settings are written into `spring`.

Impact:

- Profile switching can silently copy one profile's settings into another.
- Seasonal tuning can be lost without an obvious warning.

Possible fix plan:

1. Separate "switch active profile" from "edit selected profile settings".
2. Or reload the form immediately when the profile selector changes.
3. Or use explicit per-profile tabs/forms so the edited profile is unambiguous.

### 5. Emergency start persists override state before charger commands succeed

Severity: Medium

Evidence:

- [web_config.py](web_config.py:410)

`/api/emergency/start` calls `activate_emergency_override()` before setting mode/current/enabled on the charger.

Impact:

- If the hardware call fails, the API returns an error but the persistent automation state may already be in emergency mode.
- On the next automation poll, the controller can keep trying to force emergency charging even though the manual start request failed.

Possible fix plan:

1. Perform charger writes first, then persist emergency state only after success.
2. Or roll back the emergency flag inside the exception path.
3. Add an integration test for "charger write fails during emergency start".

### 6. Active sessions are written with `ended_at` before they actually end

Severity: Medium

Evidence:

- [solar_ev_controller.py](solar_ev_controller.py:672)
- [templates/sessions.html](templates/sessions.html:91)

`MySQLSessionStore.update_session()` updates `ended_at` on every active-session refresh using `last_seen_at`.

Impact:

- Active rows look ended in MySQL and in the sessions UI.
- `ended_at` no longer means "session finished"; it means "last observation time while maybe still active".
- Any downstream reporting that relies on `ended_at IS NULL` semantics will be wrong.

Possible fix plan:

1. Keep `ended_at = NULL` until `finish_session()`.
2. If live last-seen tracking is needed in SQL, add a dedicated `last_seen_at` column.
3. Update the sessions UI to show active/in-progress rows explicitly.

### 7. Tesla solar "share" controls are loaded and displayed but do not affect charging decisions

Severity: Medium

Evidence:

- [solar_ev_controller.py](solar_ev_controller.py:1272)
- [solar_ev_controller.py](solar_ev_controller.py:2158)
- [solar_ev_controller.py](solar_ev_controller.py:2268)
- [templates/charger.html](templates/charger.html:152)

The controller loads:

- `tesla_solar_powerwall_boost_ev_share_pct`
- `tesla_solar_powerwall_high_ev_share_pct`
- `tesla_solar_powerwall_low_ev_share_pct`
- `tesla_solar_after4pm_ev_share_pct`

But `_apply_summer_tesla_solar_policy()` does not use them when selecting `target_amps`. The actual target now comes from incremental margin logic (`solar_margin_w` against boost/trim thresholds). The share values are only echoed into telemetry.

Impact:

- The UI explains behavior in terms of EV share percentages, but the controller is making decisions from a different model.
- Even after the persistence bug is fixed, changing these share settings will not change charging behavior.

Possible fix plan:

1. Decide whether share-based control or margin-based control is the intended model.
2. If share-based control is intended, wire these fields into `target_amps` calculation.
3. If margin-based control is intended, remove or relabel the dead share controls in the UI and status output.

## Suggested Fix Order

1. Fix config persistence and profile editing semantics.
2. Fix Tesla OAuth state generation flow.
3. Restore true live-with-fallback behavior in `/api/status`.
4. Fix emergency-start state ordering.
5. Fix session DB semantics.
6. Clean up the Tesla solar control model so the UI matches real behavior.
